## 3. Оракул (определение существования галереи)

### 3.1. Метод обнаружения

Обнаружен детерминированный метод определения существования галереи **даже во время бана** IP — через размер тела HTTP-ответа (Content-Length) при запросе к `/gallery/{id}`.

### 3.2. Измеренные размеры ответов

| HTTP | Content-Length (gzip) | Content-Length (raw) | Значение |
|------|----------------------|---------------------|----------|
| 200 | 57,556–57,557B | 59,473B | Галерея существует, IP не забанен |
| 404 | **28,099B** | 28,904B (в HAR) | Галерея существует, IP в софт-бане |
| 404 | **28,073B** | 28,904B (в HAR) | Галерея не существует (или хард-бан) |
| 404 | 28,075B | — | Вариант (редкий) |
| 404 | 12,060B | — | Урезанная страница 404 (частичный бан) |
| — | 0B | — | HEAD-запрос (без тела) |

**Ключевое наблюдение:** разница между 28,099B (exists+banned) и 28,073B (never existed) составляет 26 байт. Эта разница стабильно воспроизводилась на:
- `y3tXqH0` (существует) vs `zzzzzzz` (не существует) — Осло, забаненный IP
- `XFkkFgw` (существует) vs `XFkkFga` (не существует) — Осло, забаненный IP
- `CF08KT5` — не проверялся в софт-бане (WSL AdGuard не дал нужного состояния)

### 3.3. Ограничения оракула

Оракул работает только в состоянии «софт-бан». В состоянии «хард-бан» (полный бан) ВСЕ ID возвращают 28,073B — различие исчезает. Переход между состояниями происходит при накоплении невалидных запросов: свежий IP → софт-бан (5–15 запросов) → хард-бан (15–40+ запросов).

### 3.4. 301 Redirect

Пользователь выдвинул гипотезу о 301 редиректе как механизме оракула. Проверка через HAR-файлы показала:
- 301 возникает только при запросе URL **со слешем на конце** (нормализация `/gallery/ID/` → `/gallery/ID`)
- 301 **не связан** с существованием галереи — это стандартная нормализация URL
- Галерея `XFkkFgw` со слешем → 301 → без слеша → 200 (до бана) или 404 (в бане)
- Галерея `XFkkFga` (выдуманная) → сразу 404, без 301

301 возникает при запросе со слешем для любого ID (и существующего, и несуществующего). Разница в размерах тела 404-страницы (28099B vs 28073B) наблюдается без 301 редиректа.

## 4. Эксперименты с HTTP-методами и банами

### 4.1. HEAD vs GET

**Цель:** проверить, триггерит ли HEAD-запрос бан так же, как GET.

**Метод:** на одном свежем IP выполнить N HEAD-запросов к невалидным ID, затем проверить реф через GET на `/gallery/`.

**Результаты:**

| Локация | Метод | Кол-во запросов до бана |
|---------|-------|------------------------|
| Лондон | HEAD | ~25 (бан подтверждён) |
| Париж | HEAD | ≤40 (бан подтверждён) |
| Токио | GET | ~20 (бан подтверждён) |
| Сингапур | GET | ≤40 (бан подтверждён) |

**Результат:** HEAD и GET показали схожие пороги бана (20–40 запросов, зависит от локации). Тело HEAD-ответа всегда 0B — определение существования галереи через размер тела при HEAD-запросах невозможно.

### 4.2. Перекрёстный бан (cross-endpoint)

**Цель:** проверить, распространяется ли бан с `/json` на `/gallery`.

**Метод:** триггернуть бан через невалидные запросы к `/json?action=list&album=...`, затем проверить валидный реф через `/gallery/{ref}`.

**Результат:** бан **перекрёстный**. После бана на `/json`, запросы к `/gallery/{ref}` также возвращают 404 (28099B).

### 4.3. CDN (i.testimage.cc)

Запросы к CDN (`i.testimage.cc/{id}.jpg`) возвращают 404 с 524B — это другое тело, не совпадающее ни с одним из вариантов `/gallery/`. Оракул на CDN не работает.

### 4.4. Порог бана (сводка)

| Локация | Тип бана | Порог |
|---------|---------|-------|
| Осло | GET | ~10 |
| Токио | GET | ~20 |
| Лондон | HEAD | ~25 |
| Париж | HEAD | ≤40 |
| Сингапур | GET | ≤40 |

Порог зависит от гео-локации (разные WAF-правила на разных edge-серверах). Азиатские локации (Сингапур) имеют более высокий порог.

## 10. Анализ HAR: загрузка через `/web`

Страница `https://testimage.cc/web` и JavaScript `web.js` реализуют механизм загрузки:

1. `submitRequest()` → формирует FormData с полями: `gallery`, `optsize`, `expire`, `url`/`file`, `numfiles`, `upload_session`
2. `XMLHttpRequest` → `POST https://testimage.cc/json`
3. Сервер возвращает JSON: `{"url":"...","image":"...","gallery":"ID"}`

## 16. Эволюция сканера: мульти-портовая архитектура

### 16.1. Переход от одного порта к пулу

Сессия 19.07 использовала один SOCKS5-порт (1080) с последовательной ротацией AdGuard. Пропускная способность: ~10 запросов до бана, затем смена локации (5–10 сек), итого ~1 rps.

**Решение (21.07):** запускать несколько AdGuard-инстансов на разных портах, каждый со своей локацией.

### 16.2. Скрипты настройки

| Скрипт | Назначение |
|--------|-----------|
| `adguard_pool.sh` | Pool manager на 8 инстансов (1081–1088) |
| `setup_30_ports.sh` | 30 портов (1080–1109), 30 локаций |
| `setup_60_ports.sh` | 60 портов (1080–1139), 60 локаций |
| `start_60_ports.sh` | Быстрый запуск после ребута |

**Механика:** каждый порт получает копию `~/.local/share/adguardvpn-cli` с отдельным `HOME=/tmp/proxy_port_{N}`. Это обходит ограничение AdGuard на один инстанс.

### 16.3. Версии сканера

| Версия | Файл | Архитектура |
|--------|------|------------|
| v1 | `scanner.py` | Один порт, ротация VPN |
| v2 | `scanner_v2.py` | 10 портов, round-robin |
| v3 | `scanner_v3.py` | 30 портов, адаптивный радиус |
| v4 | `scanner_v4.py` | 29 портов, 3 probes/batch, 2 chain |
| dual | `scanner_dual.py` | Два независимых сканера (A: порты 1080-1109, B: 1110-1139) |
| radius | `scanner_radius.py` | Радиусный перебор вокруг якорей из anchor_ids.txt |

**Актуальная версия:** `scanner.py` (main) — 60 портов, 6 probes (2 chain × 3) на порт перед ротацией.

### 16.4. Алгоритм сканирования (scanner.py)

```
1. Проверить все порты через check_ref(y3tXqH0) → alive_ports
2. Выбрать случайный seed из anchor_ids.txt (2207 ID)
3. Сгенерировать NUM_TARGETS последовательных ID от seed
4. Round-robin по alive_ports:
   - Перед портом: check_ref → слепой? skip
   - Chain A: 3 probes с pre/post ref-check
   - Chain B: ещё 3 probes
   - Если порт ослеп: rotate_port() → adguardvpn-cli connect -y
5. Каждые 50 проверок: progress с rps, found, rotated
```

## 24. Результаты внешнего ревью

Отчёт был передан на ревью третьему лицу. Ниже — ключевые методологические замечания и пересмотренный план исследований.

### 24.1. Проблема №1: Выводы, не поддержанные данными

| Текущая формулировка | Фактически измерено | Корректная формулировка |
|---------------------|-------------------|----------------------|
| «Генератор — HMAC-based PRNG» | Нет корреляции со временем, нет локальности, есть сессионность, высокая энтропия | «Модель совместима с: HMAC-DRBG, AES-CTR, ChaCha20, SipHash, xoshiro, собственным генератором» |
| «Оракул работает» | Наблюдалась корреляция размера с существованием на ref-примерах | «Требуется статистическая валидация: confusion matrix, precision, recall, ROC» |
| «Вокруг ID нет соседей» | Проверено ±50 вокруг 200 якорей (выборка ~2200) | «В данной выборке соседей не найдено» |
| «Локальности нет» | 618 старых кластеров мертвы | «Локальность может существовать внутри сессионного namespace, это другая гипотеза» |

### 24.2. Проблема №2: Нет валидации оракула

Оракул по размеру тела — самое интересное место исследования, но его точность не измерена.

**Что отсутствует:**
- Confusion matrix (сколько 28099/28073 для 100 известных exist + 1000 известных not-exist)
- False Positive Rate / False Negative Rate
- Precision / Recall
- ROC-кривая
- Проверка устойчивости: по странам, edge-серверам, времени, cookies, Accept-Language, HTTP/2 vs HTTP/3

**Размеры **404** ответа варьируются (28073/28075/28083/28099/12060) — возможно, классификатор не бинарный, а зависит от:**
- Локали / CF edge
- Cookies (сессионное состояние)
- A/B тестов на стороне сервера
- Версии HTML-шаблона
- Feature flags Cloudflare

**Вывод:** без confusion matrix нельзя говорить «оракул работает».

### 24.3. Проблема №3: Изменение нескольких факторов одновременно

Пример: смена IP = смена страны = смена порта = смена времени. Нельзя определить, какой фактор изменил поведение.

**Принцип:** менять один фактор за эксперимент.

### 24.4. Проблема №4: Отсутствие статистических критериев

- χ² > 80 без p-value и ожидаемых частот — ничего не говорит
- Нет bootstrap, confidence interval, Bayes factor
- «Энтропия 5.67/5.95 бит» без доверительного интервала

### 24.5. Проблема №5: Недостаточная проверка гипотезы PRNG

Упор был сделан на идентификацию конкретного алгоритма, а не на базовые свойства:
- Независимость позиций символов
- Зависимость символ ↔ следующий символ
- Bias внутри base62
- Периодичность
- Циклы
- Повторяемость после новой сессии

### 24.6. Проблема №6: Слишком быстро отброшена гипотеза последовательных ID

Факт: gap=1 в одной загрузке. Но вывод «локальности нет» сделан на основе глобальной проверки. Локальность может существовать **внутри namespace сессии** — это другая гипотеза, не проверенная.

### 24.7. Проблема №7: Не проверены арифметические паттерны вокруг находки

Находка `bv68s9M` = seed+18. Не проверены:
- seed+36, seed+54, seed+72, ...
- Gray ordering
- Bit permutation

### 24.8. Проблема №8: Масштабирование brute-force не решает проблему

Пространство 62^7 = 3.52×10^12. Увеличение числа прокси с 2 до 60 даёт прирост пропускной способности, но вероятность успеха остаётся пренебрежимо малой. Ресурсы эффективнее направить на анализ схемы генерации.

## 25. Пересмотренный план исследований

### Этап 1: Валидация оракула (приоритет: КРИТИЧЕСКИЙ)

1. Собрать выборку:
   - 100 известных существующих галерей (свои, SESSIONKEY)
   - 100 известных удалённых (создать → удалить)
   - 1000 случайных несуществующих ID
   - 100 чужих найденных
2. Для каждой — записать размер ответа через несколько разных IP/локаций
3. Построить confusion matrix
4. Вычислить precision, recall, F1
5. Проверить устойчивость по: стране, edge, времени суток, User-Agent, cookies

### Этап 2: Статистический анализ ID (приоритет: ВЫСОКИЙ)

1. Проверка независимости позиций символов (chi-square с p-value)
2. Биграммы и триграммы — сравнение с равномерным распределением
3. NIST SP 800-22 на бинарном представлении ID (frequency, runs, DFT, ...)
4. Автокорреляция по порядку генерации
5. Проверка периодичности
6. Повторяемость после новой сессии (Chrome vs Incognito — уже есть 0 пересечений, расширить)
7. Анализ префиксов и суффиксов по отдельности

### Этап 3: Проверка гипотезы сессионной локальности (приоритет: СРЕДНИЙ)

1. В одной сессии создать 50+ галерей
2. Проверить gap между ID (уже видели gap=1)
3. Проверить отсутствие gap в чужих сессиях (SESSIONKEY чтение)
4. Если gap=1 подтверждается в сессии — брутить вокруг свежих сидов в пределах той же сессии

### Этап 4: Проверка арифметических паттернов вокруг находки (приоритет: НИЗКИЙ)

1. Для `bv68s9M` (seed+18): проверить +36, +54, +72
2. Проверить Gray-code соседей
3. Проверить bit-flip соседей

### Что исключить из плана

- Перебор ±5, ±50, ±100 вокруг старых якорей (пространство слишком велико)
- Случайный brute-force (шанс попадания ~0 для 62^7)
- Дальнейшее увеличение числа прокси (diminishing returns)

## 26. Общая оценка состояния исследования

**Сильные стороны:**
- Аккуратная фиксация наблюдений
- Попытка строить гипотезы на основе измерений
- Контрольные (reference) галереи
- Повторяемость экспериментов на разных IP
- Фиксация отрицательных результатов
- Количественная статистика вместо единичных наблюдений

**Слабые стороны:**
- Гипотезы формулируются как установленные факты
- Статистическая валидация недостаточна (нет confusion matrix, p-value, доверительных интервалов)
- Отрицательные результаты трактуются как доказательства
- Изменение нескольких факторов одновременно
- Слишком быстрое отбрасывание альтернативных гипотез

**Приоритет на сейчас:** валидация оракула (Этап 1) — без этого все остальные выводы висят в воздухе.
