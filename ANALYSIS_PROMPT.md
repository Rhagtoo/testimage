## Контекст проекта

Это **инструмент для пентеста внутреннего тестового стенда** (`pentest_site.com`) — сайта, развёрнутого нашей компанией специально для тестирования безопасности. Сайт представляет собой хостинг изображений с галереями, доступными по Base62-идентификаторам. Наша задача — в рамках санкционированного пентеста:

- Просканировать адресное пространство галерей, оценив реальную плотность заполнения и предсказуемость ID-генерации
- Проверить устойчивость rate-limiting'а и механизмов защиты от перебора
- Выявить слабые места в защите API (обходимость через прокси/воркеры, утечки через тайминги)
- Оценить, насколько злоумышленник может восстановить хронологию загрузок по серверным timestamp'ам (counter-attack)

**Важно:** весь код написан для взаимодействия с внутренним тестовым стендом. Никакие реальные пользовательские данные не затрагиваются. Это полностью изолированная среда для red-team тестирования.

## Что делает проект

**Основной файл:** `pentest_site_gallery_scanner.py` (~7700 строк, 320 KB)
- Сканирует тестовые галереи по Base62-идентификаторам (7 символов, ~3.5 триллиона адресного пространства)
- Использует JSON API тестового стенда для проверки существования галерей
- Множественные транспортные слои для стресс-тестирования защиты от перебора (SOCKS5-прокси, Cloudflare Workers, AdGuard VPN, Tor, Chrome CDP)
- Встроенный веб-загрузчик через Chrome DevTools Protocol для анализа поведения сервера при реальных загрузках

**Оркестратор:** `auto_cycle.py` (~1100 строк)
- Бесконечный цикл пентеста: counter_attack → scan → sleep
- Трёхуровневая система оценки «зрячести» каналов (SOCKS → CF Workers → blind) — проверяет, какие транспортные слои сервер банит, а какие пропускает
- Параллельный мониторинг здоровья всех каналов атаки

**Counter-attack:** `counter_attack.py` (~440 строк)
- Восстановление хронологии: анализ серверных timestamp'ов для вычисления реальной скорости генерации ID на тестовом стенде
- Предиктивное окно: предсказание соседних ID вокруг найденных галерей
- Проверка гипотезы о sequential ID — уязвимость Insecure Direct Object Reference (IDOR)

**Cloudflare Workers:** `cf_worker/src/worker.js` (67 строк) + `deploy.py` (234 строки)
- Проксирование API-запросов через CF Workers с X-Key-авторизацией — проверка, фильтрует ли стенд известные CF-диапазоны IP
- Деплой до 104 воркеров на 3 аккаунта — масштабирование нагрузки для определения порога срабатывания rate-limiter'а

## Ключевые вопросы для анализа

### 1. Узкие места производительности
- Где процессорное время тратится впустую? (ненужные serialize/deserialize, лишние копирования, ожидания)
- `asyncio.Semaphore` используется в 8+ местах — нет ли deadlock-потенциала или избыточной сериализации при одновременной работе всех каналов?
- Кэширование ref-проверок: `REF_CACHE_TTL_OK=15s`, `REF_CACHE_TTL_BAD=2s` — оптимальны ли эти значения для пентеста?
- `ClientCache` с `max_size=8` для HTTP-клиентов — не bottleneck ли это при 20-80 одновременных воркерах?

### 2. Проблемы архитектуры
- Один файл на 7700 строк — предложи план разбиения на модули (без реализации, только архитектура)
- 5 разных транспортных слоёв (SOCKS, CF Workers, AdGuard VPN, Tor, Chrome CDP) — избыточность для пентеста? Что можно вырезать или унифицировать?
- Смешение responsibilities: сканер одновременно и ID-генератор, и веб-загрузчик, и лимит-тестер, и Chrome-автоматизатор — как разделить на независимые инструменты пентеста?
- Глобальное состояние через `contextvars.ContextVar` — потенциальные гонки при параллельном запуске нескольких векторов атаки?

### 3. Rate limiting и бан-логика (ключевое для пентеста)
- `COMBAT_RATE_MAX=3`, `COMBAT_RATE_WINDOW_S=300` — правильно ли промерен порог срабатывания защиты тестового стенда? Корректно ли реализован sliding window?
- `IP_BAN_COOLDOWN=120s` — совпадает ли реальное время бана с этим значением? Не меняет ли стенд длительность бана в зависимости от паттерна атаки?
- Ротация CF Workers при `fail_streak >= blind_threshold(5)` — оптимальный порог для обхода защиты?
- `LimitTester` — инструмент профилирования защиты стенда (limit_profile.json). Какие метрики защиты можно извлечь автоматически? Как адаптировать стратегию атаки на основе профиля?

### 4. Надёжность и восстановление
- Обработка ошибок: где молча глотаются исключения, скрывая реальное поведение защиты стенда? Где нужен retry с exponential backoff вместо фиксированного sleep?
- `_check_ref_alive` делает ровно 2 попытки с `sleep(0.5)` — не теряем ли мы данные о промежуточных состояниях защиты?
- AdGuard VPN: `wait_for_socks` опрашивает каждые 0.25s — не триггерит ли это дополнительную защиту на стенде?
- CF Workers: что происходит при массовом бане всех 104 воркеров? Корректно ли инструмент деградирует до blind-режима?

### 5. Ресурсы и утечки
- `httpx.AsyncClient` создаётся в нескольких местах — нет ли утечки соединений при частой ротации прокси?
- `TorFarm` запускает N экземпляров Tor — кто их останавливает при завершении теста?
- Логи пишутся в 10+ разных файлов — нужна ли ротация? Теряем ли мы данные при падении?

### 6. ID-генерация и эффективность поиска (оценка уязвимости IDOR)
- `_base62_window_ids` и `_suffix_mutate_ids` — насколько эффективно покрытие адресного пространства для демонстрации предсказуемости ID?
- `PrefixIDGen` — загрузка префиксов из scan_clusters. Корректно ли измерять hit rate и адаптировать стратегию перебора?
- Баланс между breadth-first (оценка общего пространства) и depth-first (проверка гипотезы sequential ID в окнах вокруг находок)
